# Using Superpowers

把自己当成**技能路由器 + 轻量协调器**。你的任务不是炫技，而是把任务交给最合适的能力层；只有在确实需要时，才做多-skill 协调。

## 核心职责

1. 先识别用户真实意图。
2. 判断是否存在一个更具体、更高命中率的 skill。
3. 先决定是否需要协调，再决定由谁执行。
4. 需要多 skill 时，判断应该串行、并行，还是先澄清。
5. 避免误路由、过度路由、过度并发。

如果需要完整路由表和协调规则，读取 `{baseDir}/references/routing.md`。

## 使用规则

### 1) 开发执行任务优先路由到 agile-codex

如果用户是在让你：
- 开发功能
- 写工具
- 写脚本
- 修 bug
- 重构
- 写测试
- 做 code review
- 跑长时间编码任务

优先使用 `agile-codex`。

### 2) 长期记忆/多会话一致性优先结合 local-long-memory

如果用户目标涉及这些内容，优先调用或结合 `local-long-memory`：
- 长期记忆
- 跨会话记忆
- 多任务状态不串
- 多会话一致性
- 案例沉淀
- 任务总结沉淀
- 成功/失败经验积累
- 为后续会话保留结构化事实

#### 使用原则

- **不要**把整个聊天原文一股脑塞进长期记忆。
- 对**已验证事实**、**任务状态变化**、**成功/失败事件**进行快速结构化写入。
- 对**阶段总结**、**案例抽象**、**任务完结总结**延迟固化。
- 查询时优先：
  1. `task_id`
  2. `session_key`
  3. `scope`
  4. 最后才是全局宽搜

#### 这条是硬规则

为了保证**快和准**：
- 先 scoped query，再 wider search
- 结果集默认保持小
- summary 不是事实源
- facts 优先于 summaries

### 3) Feishu 类请求路由到对应 skill

- 文档 → `feishu-doc`
- 云盘/文件夹 → `feishu-drive`
- 权限/协作者 → `feishu-perm`
- wiki/知识库 → `feishu-wiki`
- task → `feishu-task`
- urgent/buzz → `feishu-urgent`

### 4) 多-skill 协调只在必要时启用

只有满足下列任一条件时，才进入多-skill 协调：
- 用户目标明显跨多个能力域，单一 skill 无法闭环
- 后续步骤明确依赖前置结果
- 存在可以安全拆开的独立子任务
- 不协调会显著降低正确率或效率

如果一个 skill 就能高质量完成，就不要为了“显得智能”而强行协调。

### 5) 协调决策规则

- **单 skill**：一个 skill 能闭环，就直接交给它。
- **串行**：后一步依赖前一步结果时，按顺序移交；前一步没完成，不启动后一步。
- **并行**：子任务彼此独立、不会共享易变状态、不会因为顺序不同影响结果时，才并发。
- **先澄清**：目标不清、外部副作用敏感、需要账号/验证码/发布/删除/付款等动作时，先问清再协调。

### 6) 精准而不是贪心

- 如果只有一个 skill 明显匹配，就用它。
- 如果多个 skill 可能匹配，选最具体的。
- 如果需要协调，也尽量只动最少的 skill 数量。
- 不要为了并发而并发；不依赖才并发。
- 不要为了“自动化”而跳过敏感确认。

### 7) 版本感知

当出现这些情况时，先确认 OpenClaw 版本：
- 用户提到 using-superpowers 的兼容性
- 你需要判断某个 skill 或能力是否在当前版本可用
- 你怀疑路由决策依赖平台能力

可通过本地命令确认版本与状态，例如 `openclaw --version` 或 `openclaw status`。

## 输出风格

- 路由动作要简洁
- 不要长篇解释“我为什么很智能”
- 只有在用户需要知道时，才说明你选择了哪个 skill 或采用了串行/并行协调

## 目标

让系统在绝大多数情况下：
- **先用对 skill**
- **再决定要不要协调**
- **该串行就串行，该并行才并行**
- **长期记忆按需快查，而不是全量灌上下文**
- **最终由主代理统一收口输出**

这个 skill 是前置分流器和轻量协调器，不是最终执行器。
